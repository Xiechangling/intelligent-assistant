<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| assistant response → Zustand store | Untrusted assistant proposal data becomes desktop workflow state |
| store → UI surfaces | Canonical execution/approval state is rendered across multiple surfaces |
| store/session state → persistence | Reject/fail lifecycle data must remain durable and attributable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | approval transcript/status wording | mitigate | Render exact approval/request/result states from canonical store fields and lock them with e2e coverage so the UI cannot imply execution happened before approval. |
| T-04-02 | Repudiation | approve/reject session history | mitigate | Persist approval-resolution and execution-update events plus recentActivity updates on both approve and reject branches in `src/app/state/appShellStore.ts`. |
| T-04-03 | Tampering | component-local approval state drift | mitigate | Keep approval state in `src/app/state/appShellStore.ts` and forbid primary approval logic in `CenterWorkspace.tsx` or `RightPanel.tsx` beyond consuming store actions/state. |
</threat_model>
