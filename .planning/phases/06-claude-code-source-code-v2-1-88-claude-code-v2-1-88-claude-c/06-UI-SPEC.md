## Product Intent for This Phase

Phase 6 redesigns the current MVP shell into a desktop workbench that feels materially closer to official Claude Code Desktop v2.1.88 while preserving:
- local-first behavior
- Windows-first ergonomics
- single-user scope
- existing Tauri + React + Rust architecture
- explicit approval before impactful actions

### Primary UX objective
Prioritize UI and workflow alignment with the official desktop reference before deeper runtime parity.

### Experience thesis
The app must stop reading like a generic chat tool with attached utilities and instead read as a session-first coding desktop where:
- workspace context is explicit
- resumable sessions are first-class
- attachment/connection state is visible
- approvals and review are part of the same workflow
- the center workspace is the undeniable focal surface

---

## Locked vs Discretionary

### Locked
- Keep the existing five-region shell: top toolbar, left sidebar, center workspace, right panel, bottom panel.
- Center workspace remains the primary product surface.
- Session-first coding workflow takes priority over generic conversation mode.
- Startup, chooser, active session, approval, execution, and review must read as one connected lifecycle.
- Copy tone must use official-feeling lifecycle language: Ready, Connected, Attached, Working, Awaiting approval, Review ready, Failed, Needs attention.
- Approval must continue to show command intent, project/workspace context, and working directory before execution.
- Review remains local, file-based, and session-associated.
- No collaboration, sync, plugin marketplace, or direct upstream runtime transplantation.

### Implementation may decide
- Exact component/file split used to deliver the contracts below.
- Exact naming of chooser sub-panels if semantics remain consistent.
- Exact data-shape extensions in store/backend needed to support new states.
- Exact placement of secondary metadata between right panel and bottom panel.
- Exact visual polish details such as divider treatment, row density, and motion timing, as long as they stay within this contract.

## Information Architecture and Region Hierarchy

### Region priority
1. Center workspace
2. Top toolbar
3. Left sidebar
4. Bottom panel when a lifecycle event exists
5. Right panel

### Region roles

#### TopToolbar
Purpose: persistent global context strip.

Must contain:
- current mode label
- current workspace name/path summary
- current session title or chooser context
- one top-level workflow status chip
- quick access to inspector/settings

Must not contain:
- detailed approval controls
- multi-line status explanations
- dense settings forms
- review diff content

#### LeftSidebar
Purpose: navigation and recency memory.

Must contain:
- workspace list or recent workspaces
- resumable/recent sessions grouped under the active workspace concept
- clear indication of current selection
- attention markers for sessions needing approval, review, or recovery

Must not become:
- a second transcript surface
- a dense control panel
- a duplicate of center chooser content

#### CenterWorkspace
Purpose: primary work surface.

Owns:
- startup state
- workspace/session chooser
- active session surface
- transcript
- session lifecycle framing
- inline approval summary
- inline review-ready summary
- primary composer

#### RightPanel
Purpose: support inspector.

Owns:
- metadata and configuration detail
- session diagnostics/context detail
- credential/connection settings
- capability and preset settings

Must not be required to complete the primary task flow.

#### BottomPanel
Purpose: lifecycle tray.

Owns:
- structured approval detail
- execution output
- review file list and selected diff

Behavior:
- collapsed by default when nothing needs attention
- auto-expands on approval request, execution start, execution failure, or review readiness
- stays subordinate to center workspace unless the user explicitly focuses review/output

## Startup / Chooser / Active Session / Approval / Review Flow

### 1. Startup
The startup screen must immediately answer:
- Which workspace is active or restorable?
- Which session can be resumed or attached?
- Is the app ready, connected, attached, or blocked?

#### Startup states

##### A. Recovery available
Show a primary recovery card in the center workspace with:
- session title
- workspace name
- last activity time
- current model
- last known workflow state
- primary CTA: `Resume session`
- secondary CTA: `Open session chooser`

##### B. No recovery but workspace exists
Show session chooser by default for the active workspace.

##### C. No workspace selected
Show workspace-first landing state with:
- explanation that coding sessions require a local workspace
- primary CTA: `Open workspace`
- secondary affordance for conversation-only mode, visually subordinate

##### D. Recovery failure
Show blocking-but-recoverable state:
- heading: `Session recovery failed`
- body explains local session state could not be restored and user can reopen history or start fresh
- primary CTA: `Open session chooser`
- secondary CTA: `Start new session`

### 2. Session chooser
The current `project-sessions` view must evolve into a true chooser surface.

Chooser sections, in order:
1. current workspace summary
2. resumable session spotlight row if one exists
3. recent sessions list
4. action row for new session
5. optional conversation-mode entry, visually de-emphasized

Each chooser row must show:
- session title
- workspace/project label
- model label
- last activity time
- lifecycle status
- secondary summary line from recent activity
- attention marker when pending approval, failed, or review-ready

Primary row actions:
- `Resume`
- `Attach`
- `Start new session`

Contract:
- `Resume` is used for a persisted local session with transcript history.
- `Attach` is used when the UI is reconnecting to an already-active workflow context; in local implementation this may map to resume semantics, but the label should preserve the official mental model where relevant.
- `Start new session` always creates a clean session under the active workspace.

### 3. Active session
Once a session is active, center workspace switches into a session surface with four vertical strata:
1. session header
2. inline lifecycle summaries
3. transcript
4. composer

#### Session header must include
- session title
- workspace name
- model
- lifecycle state chip
- last activity time
- optional secondary line for current task/activity summary

#### Transcript contract
- conversation remains primary reading surface
- status events are visually subordinate to messages but chronologically integrated
- assistant streaming state must appear in-context at the transcript tail
- approval and review should appear both inline in transcript chronology and in dedicated surfaces

### 4. Approval flow
When the assistant proposes an impactful action:
- center workspace shows an inline approval summary card above the transcript tail
- bottom panel auto-expands to the approval tab
- toolbar status changes to `Awaiting approval`
- session row and chooser row gain attention marker

Approval detail must include:
- command summary
- exact command string
- workspace/project path
- working directory
- risk/impact label
- clear approve and reject actions

Primary CTA: `Approve and run`
Secondary CTA: `Reject command`

### 5. Execution flow
After approval:
- bottom panel switches to execution output tab
- toolbar status changes to `Working`
- session header status changes to `Working`
- transcript appends an execution start event
- output streams live without hiding transcript context

### 6. Review flow
When changed files exist:
- transcript appends `Review ready`
- center workspace shows a compact review-ready summary card
- bottom panel auto-switches to review tab
- toolbar status changes to `Review ready`
- chooser/session rows can display a review-ready marker until visited

Review surface must provide:
- changed file count
- selectable changed file list
- selected diff preview
- file-level summary text
- follow-up path back to composer for next instruction

## Page and Component-Level Contract

### AppShell
- Preserve three-row shell grid with sticky toolbar and expandable right drawer.
- Shell should continue to feel like a desktop application, not a mobile/web card stack.
- Drawer open state must not visually overpower the center workspace.

### WorkspaceChooser
Must replace the current plain session-history list behavior with a structured landing surface.

Required subcomponents:
- `WorkspaceSummaryCard`
- `RecoveryCallout` when applicable
- `ResumeSessionSpotlight` when applicable
- `SessionList`
- `NewSessionActionBar`

### WorkspaceSummaryCard
Must show:
- workspace name
- truncated path
- workspace readiness state
- last active session count or summary
- primary CTA `Start new session` when no obvious resume target exists

### SessionRow
Must include:
- title
- status chip
- recent summary
- metadata row: workspace, model, time
- optional attention badges

Interaction rules:
- single click selects/focuses row
- primary action is explicit within row or row click
- hover only increases affordance; it must not change meaning

### SessionSurface
Composition:
- `SessionHeader`
- optional `InlineApprovalSummary`
- optional `InlineReviewSummary`
- `TranscriptLog`
- `Composer`

### SessionHeader
Must present the session as a durable work context, not as a chat title bar.

Required content:
- title
- workspace/project name
- model
- workflow state chip
- last activity
- optional current activity summary

### TranscriptLog
Event types supported in Phase 6:
- user message
- assistant message
- stage status
- tool summary
- approval request
- approval resolution
- execution update
- review available
- recovery info
- connection/attachment info when added

Visual contract:
- user and assistant messages remain most visually prominent
- system events use a quieter surface treatment
- approval and failure-related events use semantic emphasis
- transcript width stays optimized for scanning; do not span full panel width

### Composer
Purpose: session continuation, not isolated chat input.

Must support:
- text prompt entry
- Claude Code-style slash command entry hints
- file/image attachments
- disabled state during streaming or while approval blocks input
- keyboard submit shortcut display/behavior

Primary CTA label:
- project mode: `Send instruction`
- conversation mode: `Send message`

### ApprovalSurface
Lives in bottom panel and optionally mirrored inline.

Required fields:
- summary
- command
- workspace path
- working directory
- approval status
- approve/reject buttons

### OutputSurface
- chronological output stream
- stream labels: system/stdout/stderr
- keep monospace for output text
- empty state must communicate waiting, not absence of capability

### ReviewSurface
Required split layout:
- left file rail
- right diff preview

Must support:
- first changed file auto-selected when review becomes available
- explicit empty state when no changed files are present yet
- review remains tied to the active session and last execution record

### RightPanel Context view
Must show only supportive context:
- mode
- workspace
- model
- credential state
- active session metadata
- approval summary when pending
- execution summary when running/completed

### RightPanel Settings view
Must retain settings ownership for:
- credentials
- endpoint/base URL
- presets
- workflow capabilities

But forms must read as compact desktop settings, not as the main task surface.

## State System and State Vocabulary

### Canonical top-level workflow states
Use these exact product states across toolbar, chooser, session header, sidebar markers, and bottom panel:
- `Ready`
- `Connecting`
- `Connected`
- `Attached`
- `Working`
- `Awaiting approval`
- `Review ready`
- `Failed`
- `Needs attention`

### Mapping contract for current implementation
Current store/runtime terms may remain internal, but UI must map them to the canonical vocabulary above.

| Current/internal signal | UI state |
|---|---|
| no active work, idle assistant | Ready |
| recovery/restore in progress | Connecting |
| credentials/service reachable but no active attached session | Connected |
| active restored/resumed session open in workspace | Attached |
| assistant streaming or approved execution running | Working |
| pending proposal exists | Awaiting approval |
| changed files available for inspection | Review ready |
| assistant error or execution failure | Failed |
| session status is `needs-attention` | Needs attention |

### Required state domains
The planner must ensure the store can represent all of the following explicitly:
- startup state
- recovery availability
- recovery in progress / success / failure
- chooser readiness
- active workspace selection
- no-workspace-selected state
- active session attachment state
- pending approval presence
- execution lifecycle state
- review readiness state
- session attention state
- conversation-mode fallback state

### Store contract changes required for planning
The Phase 6 plan may extend store/types to add or normalize:
- desktop workflow status enum for display
- chooser-specific row metadata
- session attention flags
- recovery/resume spotlight data
- connection/attachment semantics distinct from generic idle/active
- bottom-panel mode derived from lifecycle, not only manual tab state

### Session status labels
Use exact labels:
- `Attached`
- `Working`
- `Needs attention`
- `Complete`
- `Review ready`

### Chooser row secondary summaries
Pattern:
- concise noun/verb phrase, max one sentence fragment
- examples allowed by contract style: `Awaiting command approval`, `Ready to continue work`, `Review artifacts available`

### Empty-state copy
| Element | Copy |
|---------|------|
| Primary CTA | Open workspace |
| Empty state heading | No workspace selected |
| Empty state body | Open a local workspace to resume a coding session, attach to recent work, or start a new session. |

- output waiting copy: `Waiting for execution output.`
- review waiting copy: `No changed files are available yet.`
- recovery failure heading: `Session recovery failed`
- review summary CTA: `Open review`

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon adjacency, inline separators |
| sm | 8px | Tight control gaps, compact metadata rows |
| md | 16px | Default control spacing, card padding minimum |
| lg | 24px | Section padding, header spacing |
| xl | 32px | Main layout gaps, major card padding |
| 2xl | 48px | Major section breaks, startup/chooser spacing |
| 3xl | 64px | Rare page-level separation only |

Exceptions: 44px minimum touch/click target height for primary interactive controls; 56px toolbar row height; 40px composer send button; 28px compact status chips.

### Spacing enforcement notes
- Current CSS spacing tokens (`6, 10, 14, 18, 24, 32`) are not compliant with the required 4-point contract and must be normalized during implementation planning.
- Planner should treat the current spacing tokens as implementation debt and migrate visible layout spacing to the scale above.
- Vertical rhythm in center workspace should be based on 16/24/32, not mixed 14/18 values.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

### Typography rules
- Use exactly two weights across the shell: 400 and 600.
- Default body copy in transcript, metadata cards, and settings forms is 14px/1.5.
- Eyebrows, caps labels, status overlines, and compact pills use 12px/600.
- Major surface headings use 20px/600.
- Startup/chooser hero headings may use 28px/600 only in center workspace empty/startup states.
- Monospace output and command text use Cascadia Code or equivalent existing monospace stack at 12px, and must not introduce a third weight.

### Existing code note
Current shell already trends toward 11/12/13/14/20/22. Phase 6 must simplify this into the four-size contract above.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #0F1116 | App background, canvas, primary shell field |
| Secondary (30%) | #151920 | Cards, sidebar, toolbar, drawer, bottom tray |
| Accent (10%) | #7D98FF | Active session indicator, selected row border, active mode segment, primary focus ring, primary CTA, connected/attached state emphasis |
| Destructive | #E06C75 | Reject/destructive actions, failed state, blocking errors only |

Accent reserved for: active mode toggle, selected session row, attached/connected/working state chips, primary CTA, focus ring, review-open affordance, and approved workflow progression cues. Accent must not be used for every clickable element.

### Additional semantic colors
- Warning: `#D9A441` for approval-required, caution, and non-blocking warnings only.
- Success: use a restrained green token only for completion confirmation or healthy connected state if needed in execution; do not make green a dominant shell color.

### Color semantics contract
- Neutral/dark surfaces dominate the experience.
- Approval and caution use warning color, not accent blue.
- Failure and blocked states use destructive red.
- Review-ready may use accent treatment with neutral background; it must not look like an error.
- Avoid using multiple high-saturation regions simultaneously.

### Existing token alignment
These values intentionally align with the current shell token family in `app-shell.css` to reduce unnecessary visual churn while tightening semantics.

## Visual Hierarchy, Density, and Motion

### Hierarchy
- Center workspace uses the highest contrast and largest content area.
- Toolbar remains visually lighter than the center workspace.
- Sidebar and right panel use quieter surfaces and smaller typography.
- Bottom panel escalates visually only when approval/output/review is active.

### Density
Target density: desktop developer tool.

Rules:
- rows are compact but comfortably scannable
- avoid oversized chat bubbles and excessive empty canvas
- preserve breathing room in startup/chooser hero areas only
- transcript line length should remain readable and not span edge-to-edge

### Corner radius
- Small controls: 12px
- Secondary cards/rows: 14px to 16px
- Major center surfaces: 20px to 24px
- Pills: 999px

### Motion
- Use subtle 160ms-180ms transitions for hover, active, drawer open, and tray expansion.
- Avoid springy or playful motion.
- Status changes should feel calm and operational.

## Loading / Empty / Error / Attention States

### Loading states

#### Startup loading
- center workspace shows skeleton/placeholder chooser frame
- toolbar status: `Connecting`
- body copy: `Loading workspace and recent sessions.`

#### Session restore loading
- recovery card or banner shows restore progress
- toolbar status: `Connecting`
- center copy: `Restoring session context.`

#### Transcript streaming
- status: `Working`
- streaming indicator at transcript tail
- composer disabled
- last line should suggest active progress, not frozen waiting

#### Execution output waiting
- bottom panel output tab visible
- message: `Waiting for execution output.`

### Empty states

#### No workspace selected
- heading: `No workspace selected`
- body: `Open a local workspace to resume a coding session, attach to recent work, or start a new session.`
- CTA: `Open workspace`

#### No sessions in workspace
- heading: `No sessions yet`
- body: `Start a new session for this workspace, or switch workspaces to reopen earlier work.`
- CTA: `Start new session`

#### No transcript yet
Project mode:
- heading: `Start a coding session`
- body: `Describe the task for this workspace to begin an attached coding session.`

Conversation mode:
- heading: `Start a conversation`
- body: `Send a message without opening a workspace.`

#### No changed files
- heading/body in review tray: `No changed files are available yet.`

### Error states

#### Session history load error
- heading: `We couldn’t load session history`
- body: `Retry the chooser, or restart the app to reload local session data.`
- CTA: `Reload history`

#### Assistant response error
- heading: `Assistant request failed`
- body: `The session remains open. Review the error and send a follow-up instruction when ready.`

#### Execution error
- heading: `Execution failed`
- body: `Review output for the failing step, then continue the session with a follow-up instruction.`

#### Credential/connection error
- heading: `Connection settings need attention`
- body: `Update the local credential or endpoint settings before starting a new coding turn.`

### Attention states
Attention markers are required for:
- pending approval
- failed execution
- assistant failure
- review-ready state not yet opened
- recovery issue

Marker treatment:
- warning for approval
- red for failure
- accent for review-ready or attached activity

## Official Alignment Notes from Reference Snapshot

The UI language and structure for Phase 6 must preserve these reference ideas from the vendored official snapshot:
- session choosing/attachment is a dedicated surface, not an afterthought list
- discoverable/resumable sessions are first-class
- session event history is chronological and durable
- bridge vocabulary implies environment, attached session, activity, session count, reconnect/archive/heartbeat semantics
- desktop status should feel like an operational console, not a social/chat product

### Adaptation rule
The Tauri app must borrow the official mental model even where the local implementation does not expose a full remote bridge. Therefore:
- `Connected` and `Attached` are valid UI states even for local workflows
- chooser and session surfaces should hint at durable work contexts, not ephemeral chat threads
- recovery/resume should be visually promoted over blank-start creation when data exists
