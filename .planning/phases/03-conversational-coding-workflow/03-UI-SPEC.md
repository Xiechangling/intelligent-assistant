## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon spacing, micro gaps inside status pills, tight inline separators |
| sm | 8px | Compact gaps between buttons, chips, attachment tags, and row metadata |
| md | 16px | Default control padding, transcript vertical rhythm, inline card padding |
| lg | 24px | Surface padding for session cards, chooser cards, composer shell, and panel sections |
| xl | 32px | Main workspace padding and larger layout gaps between sections |
| 2xl | 48px | Hero empty-state padding and major empty/loading state breathing room |
| 3xl | 64px | Reserved for page-level separation only; do not require within normal Phase 3 session flow |

Exceptions: none. Any non-spacing constraints such as corner radius, typography size, or minimum control height must be specified in separate radii/typography/sizing rules rather than as spacing tokens.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Send instruction |
| Empty state heading | Start a coding session |
| Empty state body | Describe the task for this workspace to begin an attached coding session. |
| Error state | Assistant request failed. Review the message, then retry the instruction or continue from the same session. |
| Destructive confirmation | None in Phase 3 scope. Approval gating, reject flows, and review/diff confirmation language belong to later phases. |

Mode-specific copy contract:
- Project mode primary CTA: `Send instruction`
- Conversation mode primary CTA: `Send message`
- Project mode empty heading: `Start a coding session`
- Project mode empty body: `Describe the task for this workspace to begin an attached coding session.`
- Conversation mode empty heading: `Start a conversation`
- Conversation mode empty body: `Send a message without opening a workspace.`
- Project mode loading heading: `Loading workspace and recent sessions.`
- Project mode loading body: `Restoring session context.` or equivalent concise recovery progress copy.
- Workspace-required empty heading: `No workspace selected`
- Workspace-required body: `Open a local workspace to resume a coding session, attach to recent work, or start a new session.`

Interaction microcopy contract:
- Project composer placeholder: `Describe the next task for this workspace.`
- Conversation composer placeholder: `Send a message to the assistant.`
- Slash-command hint in project mode: `Type / to use slash commands`
- Streaming helper in project mode: `Streaming coding session updates.`
- Streaming helper in conversation mode: `Streaming conversation updates.`

## Phase 3 Scope Contract

This UI-SPEC is retrospective and strictly limited to Phase 3 only.

Included Phase 3 surfaces:
- Project mode coding command bar
- Project mode staged workflow presentation
- Project mode tool summary embedded in the transcript
- Streaming transcript behavior
- Conversation mode mainstream chat composer and chat layout
- Session context plus visible model/project context in toolbar and supporting panels

Explicitly not Phase 3 must-have:
- Approval gating
- Execution output viewer
- Review-ready and diff review surfaces
- Canonical Phase 6 desktop lifecycle and chooser redesign

If later-phase UI exists in the current codebase, treat it as out of scope for this contract unless required only as supporting context visibility.

---

## Mode Responsibilities

### Project mode
Purpose: project-task execution and coding guidance in the active workspace.

Required interaction contract:
- Use a coding command bar, not a mainstream chat textbox.
- The composer must frame the task as workspace-directed action.
- Slash-command affordance is allowed and should feel Claude Code-like, but remain lightweight.
- Transcript remains conversation-first, with workflow support embedded inline.
- Stage updates appear as their own compact event rows between user and assistant content.
- Tool summaries appear as compact supporting blocks inside the same transcript rather than as a separate log viewer.
- Session header must keep workspace, model, last activity, and session state visible without requiring the user to inspect each transcript item.

### Conversation mode
Purpose: general assistant discussion without a workspace-first coding workflow.

Required interaction contract:
- Use a standard chat composer and standard user/assistant message stack.
- Do not show staged workflow events unless the assistant experience truly needs them; default is plain mainstream chat flow.
- Keep the reading flow simple: user bubble, assistant bubble, streaming assistant continuation.
- Maintain visible model/session context in surrounding shell chrome, but do not burden each message row with project-task framing.

## Transcript Event Contract

Phase 3 supports exactly four must-have visible event types:

### 1. `user-message`
Visual contract:
- Right-aligned message bubble.
- Accent-filled surface using the existing blue gradient family.
- Label the role as `You`.
- Maximum width around 72% of transcript width.
- Allow multiline content and attachment chips underneath the body.

Interaction contract:
- Appears immediately on submit.
- Anchors the start of a project task or conversation turn.

### 2. `assistant-message`
Visual contract:
- Left-aligned neutral surface.
- Secondary dark surface with subtle border, never accent-filled.
- Label the role as `Assistant`.
- Same readable width rule as user messages.
- Primary reading block for results, explanations, and streamed completion.

Interaction contract:
- Starts as an empty/partial streaming placeholder and fills in progressively.
- Remains the primary reading flow in both modes.

### 3. `stage-status`
Visual contract:
- Left-aligned compact workflow card.
- Use eyebrow label plus one-sentence status body.
- Neutral/warning-tinted secondary surface only; do not style like terminal logs.
- Must read as progress, not as chat prose.

Interaction contract:
- Project mode only as a must-have.
- Use to show workflow steps such as request intake, analyzing workspace, preparing action, or wrapping up.
- Keep these concise and glanceable.
- Never expand into raw execution output or approval UI in this phase.

### 4. `tool-summary`
Visual contract:
- Left-aligned supporting card beneath or between assistant content.
- Include a concise header label and a short explanatory summary.
- Visually subordinate to assistant-message blocks via lower emphasis and smaller header treatment.

Interaction contract:
- Project mode only as a must-have.
- Summarize what the assistant used or did at a high level.
- Must not become a full audit log, output console, or diff surface.
- Keep to a short summary the user can scan and ignore if desired.

## Empty, Loading, and Error States

### Empty states
- Center empty-state content inside the main surface.
- Use one display heading, one body paragraph, and one next-step action when applicable.
- Conversation empty state is simpler than project empty state.
- Workspace-required state must direct the user to open a workspace and may include a secondary conversation-only entry.

### Loading states
- Use concise state headlines inside the workspace surface, not full-screen blocking overlays.
- Preferred heading: `Loading workspace and recent sessions.`
- Preferred body: short recovery/progress statement such as `Restoring session context.`
- Keep user context visible in toolbar while loading.

### Error states
- Use inline banners at the top of the workspace area.
- Error banner title should state the problem directly; body should state the next recovery action.
- Use destructive color only for the banner border/icon emphasis, not for the full canvas.
- Session history load failures should remain inside the chooser surface with a retry action.

---

## Surface and Layout Contract

- Preserve the existing five-region shell; Phase 3 extends the center workspace rather than redesigning the app shell.
- Top toolbar remains the primary always-visible context strip for mode, workspace, session, and workflow status.
- Right panel remains supportive metadata only; it does not replace the center transcript as the main work surface.
- Center workspace is the primary Phase 3 surface and contains session header, transcript, and composer.
- Composer stays pinned to the bottom of the session surface with a subtle top border and dark gradient backing.
- Transcript scrolls independently within the center workspace body.
